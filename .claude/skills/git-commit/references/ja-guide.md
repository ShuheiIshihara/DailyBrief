# git-commit スキル 日本語ガイド

このファイルは、スキルの設計思想・使い方・例を日本語で説明したものです。
スキル本体（SKILL.md）はコスト削減のため英語で記述されています。

---

## このスキルの役割分担

| 役割 | 担当 |
|---|---|
| ファイルのステージング（git add） | 人間 |
| コミットメッセージの作成・git commit | AI（このスキル） |
| プッシュ（git push） | 人間 |

---

## 呼び出し方

以下のような発言でスキルが自動的に起動します：

- 「コミットメッセージ作って」
- 「git commitしたい、ステージングは済んでる」
- 「差分からメッセージ考えて」
- 「バグ直したのでコミットしたい」

---

## AIが行う処理の流れ

1. `git status` / `git diff --staged` / `git log` でリポジトリの状態を確認
2. 差分の内容を読み込み、変更の目的・背景を理解
3. 日本語のコミットメッセージを生成（件名 ＋ 本文）
4. ユーザーに提案し、確認を取る
5. 承認後に `git commit` を実行してハッシュを表示

---

## 良いコミットメッセージの書き方

### 件名行（50文字以内）

動詞で始め、**何をしたのか（目的）** を簡潔に表現します。

| 変更の種類 | 件名の例 |
|---|---|
| 新機能追加 | ユーザー認証機能を追加 |
| バグ修正 | ログイン時のNullPointerExceptionを修正 |
| リファクタリング | 認証ロジックをAuthServiceに切り出し |
| ドキュメント更新 | READMEにセットアップ手順を追記 |
| 設定変更 | 本番環境のタイムアウト値を60秒に変更 |
| 削除 | 非推奨のlegacyAuth関数を削除 |
| 依存更新 | Reactを18.3.0にアップデート |

**良い例**: 「ユーザー一覧ページの検索フィルターを追加」  
**悪い例**: 「ファイルを変更した」「修正」「update」

### 本文（任意）

以下のときに件名の後に空行＋本文を追加します：

- 変更の**理由**が自明でないとき
- 複数の異なる目的の変更が含まれるとき
- 技術的な決断の背景を記録したいとき

### 複数ファイルにまたがる変更

ファイル名を列挙するのではなく、**変更全体を貫く目的・テーマ**を一文で表現します。

- ✅ 「リクエストログ機能を実装」（app.py・config.yaml・READMEをまとめて変更した場合）
- ❌ 「app.py、config.yaml、README.mdを更新」

## 設計メモ（このスキルを作った理由）

- **SKILL.md を英語で書く理由**: Claude への指示は英語のほうがトークン効率が良く、コストを抑えられる
- **日本語ガイドを別ファイルにする理由**: スキルの使い方や設計思想を日本語で残すことで、人間が読んだときに理解しやすくなる
- **git push を自動化しない理由**: push は取り消しが難しい操作なので、最終判断は人間に委ねる

## モデル選択に関する検討メモ

**意図**: コスト削減のため、Claude Haiku での実行を推奨する。

**根拠**: このスキルは外部情報の検索を一切必要としない。入力（git diff）はすべてコンテキストとして与えられ、出力は定型フォーマット（日本語コミットメッセージ）への変換である。Sonnet 相当の広範な知識検索能力は不要で、Haiku のコード読解・テキスト生成能力で対応できると判断している。

**注意点・品質トレードオフ**:
- 単一ファイルのシンプルな変更（バグ修正・機能追加）では Haiku で十分な品質が期待できる
- 複数ファイルにまたがる変更で「共通テーマを見抜く」処理は推論負荷が高く、Haiku では品質が落ちる可能性がある
- 日本語の自然さ・簡潔さでも Sonnet より劣る場合がある

**現在のスキルフォーマットの制約**: SKILL.md のフロントマターに `model` フィールドは存在しない。モデル指定はプラットフォーム側の設定で行う必要があり、スキルファイル自体には記述できない（2026年4月時点）。
